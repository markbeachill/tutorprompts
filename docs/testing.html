Auditing and testing

How to check whether the toolkit behaves like a learning tool rather than an answer machine — before releasing, adapting or recommending it widely.

Using a tool with a class? You probably want the much shorter deployment check in the deployment guide. This guide is for maintainers: people releasing toolkit versions, adapting the libraries, or investigating reported problems.

The method lives in two layers. These web pages explain the process step by step with copy-ready prompts. The canonical testing pack — release-stamped Markdown files — is what you download, archive and audit against; the pages never replace it. Current pack: v4.0.

What testing means here

These tests do not prove the toolkit is perfect. They collect evidence for an informed human judgement. Each test usually produces two files: a test record (what the AI did, captured by the collector) and an audit (a second AI's structured review of that record). You make the final call.

Two kinds of tool are judged by different standards. Interactive tutoring tools (WT1, WT4, AT10, RP4, RP5) should keep the student active and must not give polished final wording too early. Full review tools (WT2, WT3, WT5, ST1, ST2, ST3, RP3, SW1) are allowed a complete structured review — but not rewritten, submission-ready prose — and should switch to short interactive responses in follow-up turns.

The workflow

  1. Choose one test card from the cards file.
  2. Open a new AI chat and upload the library named in the card. Tool-behaviour cards may use a mini library or the master; launcher, routing and triage tests must use the master.
  3. Type prompt, choose the tool, and paste the test input and any follow-ups in order.
  4. When the final turn is complete, paste the output collector into the same chat and save the test record it produces.
  5. Spot-check at least one turn of the record against the live chat. For adversarial and release-gating tests, check every turn or capture manually.
  6. Open a second chat — ideally on a different model, since same-family audits tend to be generous — load the audit prompt, choose the matching audit code, and paste the record.
  7. Save the audit, make your own judgement, and complete the test log.

Critical checks

Test cards mark their most important checks with ★. A failed ★ check caps the audit rating at MAJOR ISSUE — or CRITICAL ISSUE where the card says so — so a serious boundary failure can never be averaged away by good behaviour elsewhere, and a cosmetic point can never sink an otherwise sound tool.

The model matrix and repeat runs

The libraries are designed to work on free plans, so weak-model behaviour is evidence, not noise. Run every release-gating test at least once on a strong model and once on a free-tier model, and record both. A single run is one sample from a distribution: run release-gating tests twice on the deciding model, and treat one pass plus one failure as a failure to investigate, not an average.

Comparing library versions (A/B)

Some library changes ship as hypotheses that testing must settle. To compare versions: run the same card, same inputs, on the old and new library in separate fresh chats on the deciding model (usually free-tier); collect and audit both; judge primarily on first-response strength, since drift has its own recovery mechanisms; and state the decision rule before you run. Record the comparison and the decision in the test log's A/B section.

Minimum suggested testing before release

For a small release: U1 (menu), U3 (triage, both inputs), U5 (I'm stuck), U6 (activation, on any new model or platform), A1–A4 (the adversarial set, with A2 run inside WT5), WT1 with all four turns, WT3 when correction behaviour changed, AT2's no-claim input, and one changed tool from each affected mini library — on the model matrix. For a bigger release: all universal and adversarial tests, at least one test per mini library, and the alternate-strand inputs for WT1, WT3, WT5 and ST1.

What to do if a test fails

Audit result Suggested action
PASS Safe to keep; log the result.
MINOR ISSUE Keep, but note for the next maintenance release.
MAJOR ISSUE Revise before recommending widely. Re-test the affected tool.
CRITICAL ISSUE Do not release or recommend until fixed and re-tested.
NOT TESTABLE Check setup, prompt loading, input quality, AI limits, or a failed record spot-check; repeat before judging.

How versions work

The prompt-library suite and the testing pack are versioned together and currently both stand at v4.0. Current downloads live in latest/; fixed archive copies live in numbered version folders. Each Markdown file carries a release stamp showing what it belongs to. The library's version history is published separately on this site and is no longer included inside the library file itself. The pack and the library co-define correct behaviour: a library release that changes a behaviour must change the matching test expectation in the same release, or the audit becomes a regression test for the previous version.
