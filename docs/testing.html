Testing and audit

Use this testing pack to check whether the prompt libraries behave like learning tools, not answer machines. The tests are practical, step by step, and designed for educators rather than software testers.

Download testing pack v2.10Open test cards

What changed in v2.10?

This maintenance update adds a deadline-pressure adversarial test to check whether the toolkit still protects student authorship when a student asks for urgent rewriting.

The audit still distinguishes between interactive tutoring tools and full review tools. Version 2.9 keeps the ST1 central-claim, WT5 style-review and WT3 correction-boundary regression checks. It also improves the testing workflow itself: clearer two-chat audit guidance, less ambiguous filename suggestions, a stronger SW3 test, clearer rating boundaries and a test log that records the AI tool, model and access plan used.

Interactive tutoring tools

These should keep the student active. They should ask the student to think, choose, answer, revise or attempt something. They should not give polished final wording too early.

Examples: WT1, WT2, WT4, AT10, RP4, RP5.

Full review tools

These can produce a full review. They should give structured feedback and priorities, but should not rewrite whole sections or produce submission-ready replacement prose.

Examples: WT3, WT5, ST1, ST2, ST3, RP3, SW1.

Simple testing workflow

  1. Choose one test card.
  2. Open a new AI chat.
  3. Upload the prompt library named in the test card.
  4. Type prompt and choose the named tool.
  5. Paste the copyable test input.
  6. Save the AI output using the suggested filename.
  7. Open a second AI chat. Use a separate chat for the audit so the audit is not influenced by the same uploaded library instructions as the tool being tested.
  8. Upload or paste the audit prompt.
  9. Choose the matching audit from the audit menu.
  10. Paste the saved output and save the audit response.

Plain-English testing terms

Smoke test

A quick check that the basics work: the menu appears, routing works, and the tool responds to simple input.

Regression test

A check that something you fixed has not broken again in a later version.

Adversarial test

A check using a vague, risky or boundary-pushing request, such as asking the tool to rewrite work for submission.

Suggested minimum before release

  • Run the universal menu and language-setting tests.
  • Run at least one adversarial test.
  • Run WT1, including the follow-up turn about sounding academic.
  • Run one changed tool from each affected mini library.
  • Record the result in the test log.

What if a test fails?

Audit resultSuggested action
PASSSafe to keep; log the result.
MINOR ISSUEKeep, but note for the next maintenance release.
MAJOR ISSUERevise before recommending widely. Re-test the affected tool.
CRITICAL ISSUEDo not release or recommend until fixed and re-tested.
NOT TESTABLECheck setup, prompt loading, input quality or AI tool limits; repeat before judging.

How versions work

The public site now uses three labels: the site package, the prompt-library suite, and the testing/audit pack. The current testing/audit pack is v2.10. Current public downloads live in latest/; fixed archive copies live in numbered version folders.

Each current Markdown download includes a release stamp showing which suite or pack it belongs to, plus its public and archive paths.
