Deployment routes

How to load the toolkit on each platform, and what each route risks.

This is the second page of the deployment guide. Choosing and setting up an AI helps you decide which system and model to use; this page covers how to load the toolkit once you have decided; the Deployment check is the short test you run on the result.

One rule covers every route on this page: whichever route you choose, run the deployment check on it before using it with students. Platforms differ, change without notice, and load instructions in different ways. The check takes ten to fifteen minutes and tests the setup students will actually meet.

Route 1 — Upload the file in a chat

The simplest route, and the one the student help pages assume. Start a new chat, upload the library file (master, mini library, or single tool), and type prompt. The menu should appear.

Best for: one-off activities, free plans, first trials. Risks: students must repeat the upload each chat; long chats on free plans can hit limits; some models summarise an uploaded file instead of activating it — exactly what the first step of the deployment check looks for.

Route 2 — Paste a single tool

Where uploads are limited or the mini library is too large for the plan, students can paste a single-tool download directly into the chat and follow it with their writing.

Best for: free plans, phones, locked-down platforms without file upload. Risks: no launcher or routing, so name the exact tool in your activity instructions; very small context limits can still truncate a pasted tool.

Route 3 — Projects, workspaces and notebooks

Platforms with Projects (ChatGPT, Claude), Gems-adjacent workspaces, or Copilot Notebooks can keep the toolkit file and related chats in one place, so the student sets up once and reuses it.

Best for: repeated individual use across weeks; tutors who want a standing test space. Risks: availability varies by plan, so keep this optional for students; project-level instructions and files may be handled differently from a fresh upload, so check on the project itself, not in an ordinary chat.

Route 4 — Shared custom AIs: custom GPTs, Gems and similar

A shared custom AI gives students a ready-made entry point: they open it and type prompt, with no file handling. The setup decisions and the pros and cons are covered on the previous page; this section covers the loading mechanics, because there are two genuinely different ways to load the toolkit into a custom AI, and they can behave differently.

Instructions field versus attached knowledge file

Instructions field. The toolkit text is pasted into the custom AI's instruction or system field. The whole library is then present every time. This is the stronger route — but instruction fields have size limits, and the master library will usually exceed them. This is where the mini libraries and single tools earn their keep: a mini library in the instructions field is generally more reliable than the master library attached as a file.

Attached knowledge file. The toolkit file is attached to the custom AI as knowledge. This fits any size — but on some platforms, attached files are searched and retrieved in fragments rather than held in full. The practical consequence: a tool can run while the global rules — the academic-integrity boundary, the teaching loop, the privacy note — are simply not present in the response. The toolkit looks like it is working and is quietly degraded. If you must use an attached file, the deployment check's boundary probe matters most, because it is the test most likely to expose missing global rules.

Platform limits and retrieval behaviour change without notice, so this page deliberately avoids stating current numbers. The honest position is: the routes differ, the platforms change, and the deployment check on your own setup is the only answer that stays true.

Keeping a shared custom AI honest

Treat a shared custom AI as a maintained teaching resource. Re-run the deployment check when the toolkit is updated, when the platform announces (or silently makes) a model change, and at the start of each term. Record the date and result somewhere colleagues can see. Make clear to students that the custom AI is a convenience, not a guarantee: the usual judgement about AI output still applies.

Institutional platforms: Copilot and Gemini at your institution

Where students use an institutional Copilot or Gemini, the available models, file handling and data arrangements are set by the institution, and may differ from the public versions of the same products. Local rules come first. Run the deployment check inside the institutional environment, not on the public product, and ask your IT or digital education team about data governance if student writing will be pasted in.

After loading: run the check

Whichever route you used: run the deployment check. It confirms the menu appears, the tool you plan to use behaves like a tutor, and the boundary holds — on your platform, your plan, and your model, today.
