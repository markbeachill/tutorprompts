Producing documentation

How to capture an AI tutoring session and turn it into something you can keep, share or hand out.

Why document at all

Three classroom situations call for a record of an AI session rather than a memory of one.

Worked examples for teaching. A good session — a student's sentence genuinely improving through diagnosis, example and attempt — is teaching material. Captured and lightly anonymised, it shows the next cohort what using the tool well actually looks like, which lands far better than a description of it.

Evidence of process. Where students declare AI use (see the SW3 AI-Use Record tool and your course's rules), a saved transcript of the session is the strongest possible companion to the declaration: it shows what the AI did and what the student did, which is the question integrity processes actually ask. A declaration says "AI gave feedback on my draft"; a transcript shows the feedback and the student's own revision.

Problem reports. When a tool misbehaves on your platform, a captured session is the difference between "it didn't work" and a report a maintainer can act on.

Capturing a session: the collector

The simplest capture is one paste. At the end of the session — never before or during it — paste the toolkit's output collector into the same chat. It produces a single record: the session details and a verbatim transcript of every turn. Copy it once and save it as a Markdown (.md) or text file.

The collector prompt, with its instructions, is on the collector page of the testing guide. One honesty note travels with it: the record is written by the same AI that ran the session, so glance over it against the chat before relying on it — especially if the session went wrong, which is exactly when the AI is the least reliable narrator of its own behaviour.

Manual capture — selecting and copying the turns yourself — remains the most reliable method, and is worth the effort for anything contentious.

Presenting a record

A saved Markdown record is already readable, searchable and easy to store alongside module materials. For something more presentable:

Paste into Word. Most AI chat interfaces preserve formatting when copied into a Word document, which is enough for a handout or a staff-development example. Tidy the headings, anonymise names, and add a short framing note saying what the example shows.

Convert to a web page. The separate AI Chat to HTML project converts a chat transcript into a clean standalone web page, and its Word page covers producing a Word-friendly version. It is an independent tool with its own setup, so treat it as an option for those comfortable with it rather than a required step — the Markdown record and the paste-into-Word route cover most classroom needs.

Anonymisation and care

Before sharing any session beyond its student: remove names, student numbers, and anything identifying in the writing itself; remember that a student's draft is their work, so ask before using it as an example, even anonymised; and for placement-related writing (nursing, education, social work), check that no patient, pupil or client details survive in the extract — the same privacy rule the toolkit itself applies.

A simple classroom pattern

For a module using the toolkit regularly, the lightweight habit is: students end each AI session by saving a collector record into a module folder alongside their SW3 record; the tutor keeps one or two good anonymised examples per term for teaching; and anything that misbehaves gets its record sent on via the About page. Documentation then accumulates as a by-product of use, instead of being reconstructed when someone asks for it.
